# Context: SOCKS5 Proxy → WireGuard → Latvia

## Проблема: почему не SSH туннель

Изначально использовался `ssh -D 1080` как SOCKS5 прокси.

**Почему SSH плохо работает под нагрузкой:**
- Каждое соединение = криптографическое рукопожатие = нагрузка CPU
- SSH не чистит зависшие каналы — они накапливаются  
- При 20-30 одновременных вкладках у нескольких пользователей CPU → 100%
- Решение только перезагрузкой

---

## Почему 3proxy

3proxy — специализированный прокси-сервер:
- Многопоточный, держит тысячи соединений без роста CPU
- SOCKS4, SOCKS5, HTTP proxy
- ~3MB RAM в простое
- Логирует каждое соединение

**Коды в логах 3proxy:**
| Код | Значение |
|-----|----------|
| `00000` | Успех |
| `00004` | Нет подходящего правила (access denied) |
| `00100` | Ошибка соединения с целевым хостом |

---

## Почему авторизация отключена

Chrome и все Chromium-браузеры **не поддерживают** аутентификацию через SOCKS5.
Это ограничение браузера, не 3proxy.

Защита строится через:
- Нестандартный порт (43473 вместо 1080/3128)
- Опционально: `ufw allow from <IP> to any port 43473`

---

## Почему WireGuard

| | SSH туннель | WireGuard |
|---|---|---|
| Шифрование | Userspace (OpenSSH) | Ядро Linux |
| CPU нагрузка | Высокая | Минимальная |
| Задержка | +5-15ms | +1-3ms |
| Восстановление после обрыва | Требует перезапуска | Автоматически |

WireGuard работает как сетевой интерфейс (`wg0`) в ядре. Пакеты маршрутизируются через него как через обычный интерфейс.

---

## Как работает маршрутизация

### На российском сервере

```
default via wg0          ← весь трафик → Латвия
10.0.0.0/24 dev wg0     ← WireGuard подсеть
```

Клиент → 3proxy → запрос уходит через `wg0` → выходит в Латвии с латвийским IP.

### Проблема SSH при AllowedIPs = 0.0.0.0/0

WireGuard захватывает весь трафик → SSH тоже уходит в туннель → SSH обрывается.

**Решение — PostUp правило:**
```
PostUp = ip rule add from 185.23.35.191 table main
```

Ядру: "трафик ОТ `185.23.35.191` (сам сервер) → основная таблица, не WireGuard".  
SSH отвечает со своего IP → соединение сохраняется.

### На латвийском сервере

```
PostUp = iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
```

Заменяет IP источника пакетов с `10.0.0.2` (WG адрес RU) на внешний IP Латвии.
Интернет видит латвийский IP.

## Адресация WireGuard

```
10.0.0.1/24  — LAT сервер (WireGuard интерфейс)
10.0.0.2/24  — RU-SOCKS сервер (WireGuard интерфейс)
```

Приватная подсеть только внутри туннеля.

## Файлы конфигурации

### RU сервер
| Файл | Назначение |
|------|-----------|
| `/etc/wireguard/wg0.conf` | WireGuard туннель к Латвии |
| `/etc/wireguard/private.key` | Приватный ключ (не передавать!) |
| `/etc/wireguard/public.key` | Публичный ключ (для LAT сервера) |
| `/etc/3proxy/3proxy.cfg` | Точка входа 3proxy |
| `/etc/3proxy/conf.cfg` | Настройки: порт, auth none |
| `/var/log/3proxy/` | Логи соединений |

### LAT сервер
| Файл | Назначение |
|------|-----------|
| `/etc/wireguard/wg0.conf` | WireGuard (exit node) |
| `/etc/wireguard/private.key` | Приватный ключ |
| `/etc/wireguard/public.key` | Публичный ключ (для RU сервера) |

## Типичные проблемы

### SSH оборвался после запуска WireGuard
Причина: `AllowedIPs = 0.0.0.0/0` без `PostUp` правила.  
Решение: перезагрузить через панель хостинга, добавить PostUp.

### 3proxy: код 00004 в логах
Причина: `auth strong` активен, Chrome не передаёт логин/пароль.  
Решение: убедиться что в conf.cfg стоит `auth none`.

### 3proxy не ставится через apt
Пакета `3proxy` нет в стандартных репозиториях Ubuntu 22.04+.  
Скрипт устанавливает через готовый .deb с GitHub или собирает из исходников.

### Туннель поднялся, но пинг между серверами не проходит
Причина: неправильные публичные ключи или несовпадение адресов.  
Решение: `wg show` на обоих — должны видеть друг друга как peer с handshake.

### curl ifconfig.me показывает российский IP
Причина: нет MASQUERADE на LAT или `AllowedIPs != 0.0.0.0/0`.  
Решение: проверить конфиги и `systemctl restart wg-quick@wg0`.
